Web 课程设计报告

Catmunity 猫咪社区管理平台

设计与实现

前后端分离 · Vue 3 + TypeScript + Vite

移动端（Vant 4） · 管理端（Ant Design Vue 4）

&#x20;摘要

Catmunity 是一个面向猫咪爱好者的社区管理平台，围绕猫咪档案管理、社区互动、AI 智能问答、猫咪打卡与后台运营管理五大场景展开。系统采用前后端分离架构：前端基于 Vue 3（组合式 API）、TypeScript、Vite 构建，移动端使用 Vant 4 组件库，管理端使用 Ant Design Vue 4 组件库，并借助 unplugin-auto-import 与 unplugin-vue-components 实现组件与 API 的按需自动引入；数据可视化使用 Apache ECharts 6，AI 对话内容渲染使用 Marked，移动端像素适配使用 postcss-px-to-viewport。项目通过 pnpm 进行包管理，前端源码组织为 views / components / api / utils / mock / styles 等分层目录，配合统一的 Axios 请求封装、路由守卫和 Mock 拦截体系，构成了一个结构清晰、可独立演示的中型 Web 应用。

本报告按照软件工程的一般流程展开，依次完成痛点与可行性分析、开发估算、需求分析、系统设计（架构设计、数据流设计、用例设计）、数据库设计、系统实现、系统测试等内容。在需求分析与系统设计部分，本报告针对系统中的 33 个核心功能点逐一给出详细的用例规约（用例名称、参与者、用例目标、前置条件、后置条件、基本流程、备选/异常流程），并绘制了系统架构图、请求处理流程图、16 类核心业务时序图与 4 张类图，力求以“图文对照代码”的方式呈现系统的设计与实现细节。

关键词：Vue 3；TypeScript；Vite；前后端分离；Vant；Ant Design Vue；用例分析；SSE 流式问答；课程设计

1  前言

Catmunity 是一个面向猫咪爱好者的社区管理平台，围绕“猫咪档案管理、社区互动、AI 问答、猫咪打卡和后台运营管理”五类核心场景展开设计。项目采用前后端分离架构，前端使用 Vue 3、TypeScript 和 Vite 构建，结合 Vant 4 与 Ant Design Vue 4 分别实现移动端与管理端界面，以满足不同终端的使用需求。从仓库结构看，前端工程位于 frontend 目录下，采用单一 package.json 管理依赖，通过 pnpm-lock.yaml 锁定版本，后端接口与部署相关内容位于 backend 目录，项目根目录另有《Catmunity 功能文档》对页面与模块功能做了详细的中文说明，本报告在写作中与该文档保持了口径一致，并在此基础上补充了更细粒度的用例分析与图示说明。

从课程设计的角度看，本项目具有较强的综合性：既包含登录注册、分页查询、表单校验等基础业务，也包含评论树递归渲染、SSE/分块流式问答、ECharts 图表展示、自定义文件上传与响应式数据统计等较复杂的交互，因此适合作为 Web 应用课程报告的完整案例。项目一共梳理出 33 个可独立验证的功能点，覆盖用户端 17 个、管理端 16 个，本报告将逐一给出规范的用例规约，并配合系统架构图、时序图和类图，完整呈现从需求到实现的设计脉络。

与此同时，项目还体现了前端工程化开发的常见流程：路由拆分与导航守卫、基于 localStorage 的登录态与角色区分、Axios 请求/响应拦截器的统一封装、组件复用（PostCard、CommentItem）以及基于请求拦截器的 Mock 数据联调机制。这些内容不仅能够满足课程设计的演示需要，也能够较完整地展示一个中型 Web 应用从工程搭建到业务落地的实现思路。

2  痛点分析、竞品分析与可行性分析

2.1 需求痛点

在现实场景中，猫咪爱好者通常需要记录猫咪成长信息、管理健康事件、分享日常动态以及获取养护建议，但现有信息往往分散在多个平台中，存在以下问题：

●	猫咪档案和健康记录分散保存，不利于长期追踪，例如疫苗接种、驱虫、绝育、体检、疾病治疗、美容护理等事件类型缺少统一记录入口。

●	普通社区平台偏重内容浏览，缺少针对宠物垂直场景的档案关联能力，帖子与具体某只猫的健康、成长信息互不关联。

●	用户缺少统一入口来查询档案、打卡、互动和咨询，操作路径分散在不同工具中。

●	管理端缺少面向猫咪业务的数据看板和维护工具，无法直观了解用户增长、内容规模和待办健康提醒。

2.2 竞品分析

与一般宠物论坛或社交平台相比，Catmunity 的差异在于将“档案管理、社区互动、健康提醒、AI 问答和打卡同步”整合到同一系统中，既兼顾内容交流，也兼顾实际管理需求。项目在实现上体现了这一差异化定位：CatArchivePage 中猫咪详情弹窗直接嵌入健康事件时间线，CameraPage 打卡弹窗提供“同步到社区”开关（syncToPost 字段），使打卡记录可以一键转化为社区内容，从而把工具型记录与内容型社区在数据层面打通。与单纯的内容社区相比，本项目更强调结构化信息管理；与单纯的工具型宠物记录应用相比，本项目又增加了互动和内容沉淀能力。

2.3 可行性分析

技术可行性：项目使用 Vue 3、TypeScript、Vite、Vue Router 4、Axios 和 ECharts 6 等成熟且社区活跃的技术栈，配合 unplugin-auto-import / unplugin-vue-components 减少样板代码，适合实现复杂交互和数据展示。

经济可行性：前端框架与开源组件库（Vant、Ant Design Vue、ECharts、Marked 等）均可免费使用，开发成本主要来自人力投入。

操作可行性：用户端采用底部导航加弹窗交互，学习成本较低；管理端采用左侧菜单加顶部栏的常见后台布局，符合管理人员操作习惯。

进度可行性：功能模块可按登录、档案、社区、问答、打卡和后台管理逐步开发，且各页面组件相对独立（如 PostCard、CommentItem 均为可复用单文件组件），便于并行开发与控制课程设计周期。

3.3 风险与对策

本项目的主要风险在于页面数量较多、交互细节较复杂，以及 AI 问答的流式接口和文件上传等能力对接口稳定性要求较高。观察 aiApi.sendMessage 的实现可以发现，项目在真实开发中确实遇到过“后端返回内容类型与预期不一致”的问题：代码里专门判断了响应 Content-Type 是否为 text/event-stream，如果不是，则按照原始分块传输流（每个 chunk 直接作为增量文本）处理，而不是强行按 SSE 的 data: 前缀解析，这种“先探测、再分流处理”的写法本身就是应对接口不确定性的一种风险对策。对应的整体策略是：先完成路由和数据类型（api/types.ts）设计，再逐步实现各页面，并使用 mock/index.ts 中的拦截器降低对后端接口的强依赖，从而保证课程设计阶段能够按时完成并可独立演示。

4  需求分析

本系统面向两类主要用户：普通用户和系统管理员。普通用户主要完成猫咪档案浏览、社区发帖评论、AI 问答、猫咪打卡和个人资料管理；管理员则负责猫咪档案维护、健康事件管理、知识库维护以及用户管理。两类角色的登录态分别存储在 localStorage 的 token / userId 与 token / userId / userType / identifier 字段中，并通过路由路径前缀（是否以 /admin 开头）加以区分。

4.2 非功能需求

除了功能需求之外，系统还需要满足以下非功能要求，且均在代码中有具体落地方式：

●	可用性：界面操作流程清晰，关键按钮有明确反馈（如 showSuccessToast / message.success 提示），减少用户学习成本。

●	性能：列表页面支持分页加载（PageHelper 风格的 pageNum/pageSize/total/pages 结构），避免一次性渲染过多数据导致卡顿。

●	安全性：登录后通过请求拦截器自动在 token 请求头注入身份信息，配合路由守卫 router.beforeEach 控制访问权限，未登录用户自动跳转登录页；后端返回 401 时同样会触发 redirectToLogin。

●	可维护性：采用 views / components / api / utils / mock / styles 分层目录和统一的 http.get/post/put/delete 封装方式，便于后续扩展和修改。

●	兼容性：移动端和桌面端分别使用 Vant 4 与 Ant Design Vue 4，并通过 postcss-px-to-viewport 的 exclude 配置将 /admin/ 目录排除在 vw 转换之外，从而做到“同一工程、两套适配策略”。

●	扩展性：预留知识库（KnowledgeBasePage）、AI 问答（QAPage / aiApi）和管理后台等模块，其中知识库模块的字段解析本身就做了“智能响应格式适配”（支持 array、{rows}、{list}、{data}、{records} 等多种响应形状），体现了对后端接口尚未完全稳定场景的兼容设计。

5  系统分析与设计

5.1 总体架构

系统采用典型的前后端分离模式。前端通过 utils/request.ts 中封装的 Axios 实例统一访问后端接口（baseURL 默认取环境变量 VITE\_API\_BASE\_URL，未配置时回退为 /api），使用路由守卫控制登录状态，并按照 views/user 与 views/admin 两个目录划分用户端与管理端页面结构。用户端更强调移动设备上的交互体验，管理端则侧重信息密度和操作效率。图 5-1 展示了系统的总体架构。

&#x20;

图 5-1  系统总体架构图

从工程组织上看，项目将页面、接口、工具、mock 数据和样式进行分层管理：views 层负责呈现和交互，api 层（index.ts + types.ts）负责定义接口方法与数据类型，utils/request.ts 负责统一请求、拦截与错误处理，mock 层（mock/index.ts + mock/data.ts）负责开发阶段基于请求拦截器的数据模拟。这种分层方式便于后续扩展和维护，也便于在后端接口未完全就绪时独立推进前端开发。

5.3 路由设计

前端路由（router/index.ts）按照功能场景划分为用户端与管理端两大部分。用户端包括登录、注册、首页框架（HomePage 作为带底部 Tab 的布局壳，默认重定向到 /archive）以及猫咪档案、社区、问答、打卡、个人中心等子页面；管理端包括后台登录页（/admin/login，独立于 AdminLayout 之外）和仪表盘、猫咪档案、健康事件、知识库、用户管理等子路由（默认重定向到 /admin/dashboard）。

路由守卫 router.beforeEach 对未登录访问进行拦截：白名单路径为 /login、/register、/admin/login，其余路径若 localStorage 中不存在 token，则根据目标路径是否以 /admin 开头分别跳转到 /admin/login 或 /login。该设计保证了不同角色的访问边界，也使页面组织更加清晰。图 5-3 展示了该守卫逻辑的判定过程。

5.4 交互设计

用户端以底部 Tab 栏（HomePage.vue 中的 5 个 tab：档案 / 社区 / 问答 / 打卡 / 我的）作为主导航，适配移动设备操作习惯；管理端采用左侧边栏 + 顶部栏布局（AdminLayout.vue），适合桌面端进行密集操作。两端在视觉语言上保持统一的品牌风格（主色为橘猫橙 #ff8c42，辅以奶油白与薄荷绿），但在组件体系上分别选择了更适合场景的 UI 框架，色彩变量统一定义在 src/styles/variables.css 中，并针对暗色模式（prefers-color-scheme: dark）做了独立的变量覆盖。

其中，用户端交互强调“单手操作”和“弹窗完成关键动作”，例如发帖（van-popup 底部弹层）、打卡、查看详情和编辑资料都在 van-popup 组件中完成；管理端则强调“表格 + 筛选表单 + 弹窗/抽屉”的典型后台交互模式（a-table + a-modal/a-drawer），以提高批量管理效率。

●	系统：codeSending 或 countdown>0 时直接 return，不重复发起请求

6  数据库设计

由于前端 api/types.ts 中已经明确定义了各业务对象的字段结构，因此数据库设计以这些 TypeScript 接口为蓝本进行反推，围绕用户、猫咪档案、健康事件、帖子、评论、打卡记录、知识库文件和 AI 会话等实体展开，并按照关系型数据库的规范补充主键、外键、索引与约束设计。

6.1 概念结构设计（E-R 关系）

系统的实体关系可以概括为：一个用户可以发布多条帖子、创建多条打卡记录、拥有多个 AI 会话、上传多个知识库文件（均为 1:N 关系）；一只猫咪档案可以关联多条健康事件和多条打卡记录（1:N）；一条帖子可以拥有多条评论，且评论支持自关联的父子嵌套结构（1:N 自关联）。具体实体关系如下：

实体A	关系	实体B	说明

用户 User	1 : N	帖子 Post	一个用户可发布多条帖子

用户 User	1 : N	打卡记录 CatCheckin	一个用户可产生多条打卡

用户 User	1 : N	AI会话 Session	一个用户可拥有多个会话

用户 User	1 : N	知识库文件 DatasetFiles	一个用户可上传多个文件

猫咪档案 CatProfile	1 : N	健康事件 CatHealthEvent	一只猫可有多条健康事件

猫咪档案 CatProfile	1 : N	打卡记录 CatCheckin	一只猫可被多次打卡

帖子 Post	1 : N	评论 Comment	一条帖子可有多条评论

评论 Comment	1 : N	评论 Comment	父评论可有多条子回复（自关联）

打卡记录 CatCheckin	1 : 1（可选）	帖子 Post	打卡若同步则生成一条关联帖子

6.3 物理结构设计要点

●	所有主键均采用 BIGINT 自增，便于分布式场景下后续迁移为雪花算法ID。

●	photo\_urls、images、tags 等数组类字段建议采用 JSON 类型存储（MySQL 5.7+支持），或拆分为独立的关联表以支持更复杂的查询。

●	点赞/收藏/打卡等具有“防重复”语义的操作，均在数据库层面建立唯一约束，与前端的 interacting 状态锁形成应用层+数据层的双重防护。

●	next\_remind\_date、created\_at（信息流排序）、parent\_id（评论树查询）等高频查询字段均建议建立普通索引。

●	涉及金额、体重等小数字段使用 DECIMAL 类型而非 FLOAT，避免精度问题（如 CatProfileVO.weight）。

7  系统实现

7.1 用户端实现

用户端围绕登录注册、猫咪档案、社区广场、AI 问答、猫咪打卡和个人中心六个页面展开，全部位于 src/views/user 目录下，对应第 5.6.1 节中功能点 1\~17。其中，社区模块支持帖子发布、点赞收藏与评论树展示；AI 问答模块支持 Markdown 渲染和流式响应；打卡模块则将日历统计、照片上传和同步社区整合到一个弹窗流程中。

7.1.1 登录注册页面（对应功能点1、2）

登录页（LoginPage.vue）和注册页（RegisterPage.vue）负责建立系统入口。登录成功后调用 authApi.login，将返回的 token 和 userId 保存到 localStorage，并跳转至 /profile；注册模块（authApi.register / sendRegisterCode）包含账号、邮箱、密码、确认密码与验证码校验，其中密码规则要求长度 8-16 位且必须同时包含字母和数字，验证码发送后有 60 秒倒计时防止重复发送。为了提高可用性，密码可见性切换（👁 / 🙈 图标）、验证码倒计时和注册成功后的自动登录跳转等交互都被纳入页面实现中。

7.1.2 社区与评论模块（对应功能点7\~12）

社区页面（CommunityPage.vue）使用信息流卡片展示帖子，支持按话题标签搜索（communityApi.getByTag）、点赞收藏（communityApi.like/unlike/favorite/unfavorite）、评论和发布（communityApi.createPost）。点赞与收藏操作均使用一个 interacting 状态集合（Set<number>）标记正在请求中的帖子 ID，防止用户重复点击导致的竞态请求；后端针对“重复点赞/收藏”返回的业务错误（消息中包含“已收藏”“未收藏”“已点赞”“未点赞”等关键词）会在 utils/request.ts 的响应拦截器中被识别为“状态同步类错误”而静默处理，不弹出 Toast 打扰用户。

评论区（CommentItem.vue）采用递归组件渲染父评论与子回复，支持通过 @提及 回复他人，并支持删除自己的评论。删除评论时，CommunityPage.vue 中的 countWithChildren 与 removeCommentById 两个递归函数会先统计被删除评论及其全部子评论的数量，再从评论树中移除对应节点，并据此把帖子的 commentCount 计数一并扣减，从而保证前端展示的评论总数与实际树结构保持一致。该模块体现了典型的“列表展示 + 详情弹层 + 表单提交 + 递归数据结构处理”组合。

7.1.3 AI 问答模块（对应功能点13、14）

AI 问答页面（QAPage.vue）维护会话列表（aiApi.getSessions）与聊天窗口，消息采用流式输出方式逐步展示，提升了交互自然度。核心实现 aiApi.sendMessage 使用原生 fetch + ReadableStream 而非 Axios 发起 POST 请求（因为 Axios 难以处理流式响应体），并通过 AbortController 支持随时中断请求。值得说明的是，该方法在解析响应时会先读取 Content-Type：若响应为标准的 text/event-stream，则按 SSE 规范以 \\n\\n 分隔事件、解析 data: 前缀并识别 \[DONE] 结束标记；若不是（后端实际返回的是 text/html;charset=utf-8 的分块传输文本），则直接将每个 chunk 解码后作为增量文本回调，这种“先探测协议、再选择解析策略”的兼容写法保证了前端在后端实现细节发生变化时仍能正常工作。系统同时支持 Markdown 渲染（marked.parse，并对 <script>/<style> 标签及 on\* 事件属性做正则过滤以降低 XSS 风险）、附件发送、停止生成（同时中断前端流并调用 aiApi.stopMessage 通知后端）和会话删除，使页面兼具实用性与演示效果。

7.1.4 打卡与个人中心（对应功能点15\~17）

打卡页面（CameraPage.vue）使用月历展示签到情况（根据 checkinDates 计算每个日期是否命中），并提供猫咪选择器（van-picker）、照片上传（userApi.uploadCheckinPhoto）和备注填写等操作，同时展示当月打卡次数、连续打卡天数与是否满勤三项统计；个人中心（ProfilePage.vue）则集中展示用户资料、统计信息（发帖数/关注数/粉丝数）和收藏内容（支持分页加载更多），并提供编辑资料弹窗（可修改账号与密码），形成完整的个人操作闭环。

7.2 管理端实现

管理端主要通过仪表盘、猫咪档案管理、健康事件管理、知识库管理和用户管理完成运营类任务，全部位于 src/views/admin 目录下并统一嵌套在 AdminLayout.vue 布局壳内，对应第 5.6.2 节中功能点 18\~33。其中，仪表盘使用图表展示总览数据，档案和健康事件模块强调查询筛选与增删改查能力，知识库模块则体现文件上传与分页管理能力。

7.2.1 仪表盘页面（对应功能点20\~23）

仪表盘（DashboardPage.vue）通过 6 个概览卡片（总用户数、7 日活跃用户、帖子总数、猫咪档案数、打卡总数、待审核内容）、ECharts 折线图（用户增长趋势，支持日/周/月维度切换与 7/14/30 天范围选择，带渐变面积填充）、ECharts 环形图（猫咪品种分布，最多展示 10 种配色并支持滚动图例）和健康事件提醒网格（展示未来 7 天内到期的事件，并按剩余天数标记“今天/明天/N 天后”与紧急程度样式）构成信息总览区。健康事件提醒列表在渲染前会额外批量拉取一次猫咪列表（pageSize 200）以构建 catProfileId 到猫咪名称的映射，从而在提醒卡片上直接展示猫咪名字而非仅展示 ID。此类页面在课程设计中有较强的展示效果，也能够体现 ECharts 与响应式布局（window resize 时调用 chart.resize()）的结合使用。

7.2.2 档案与健康事件管理（对应功能点24\~26）

档案管理页面（CatProfilePage.vue）支持按名称/品种模糊搜索、性别与状态下拉筛选、创建日期区间筛选，新增/编辑弹窗中的照片上传采用 Ant Design Vue 的自定义上传（customRequest），上传前会校验图片类型与 5MB 大小限制，上传成功后把服务端返回的 URL 写回文件项的 url 字段；删除操作在“删除当前页最后一条数据”时会自动将分页回退一页，避免出现空白页。健康事件页面（CatHealthEventPage.vue）则围绕事件类型、标题、日期和提醒日期进行维护，猫咪选择采用远程搜索下拉（每次最多加载 20 条，按输入关键字实时模糊查询），选中猫咪后自动触发该猫咪健康事件列表的刷新。两个模块都以表格和弹窗为核心交互，适合演示 CRUD 能力。

7.2.3 知识库与用户管理（对应功能点27\~33）

知识库模块（KnowledgeBasePage.vue）负责文件列表、上传和下载：由于后端返回的数据集/文件分页结构在开发阶段尚未完全固化，代码中对分页字段做了“智能响应格式适配”的兼容处理；文件下载通过 Blob + URL.createObjectURL 方式触发浏览器保存。用户管理模块（UserManagementPage.vue）负责查询、封禁、解封和角色分配：封禁需填写必填的封禁原因（≤200 字），角色分配支持超级管理员/普通管理员/普通用户三种类型切换，且封禁/解封/分配角色成功后如果当前抽屉正打开着同一用户的详情，会额外重新拉取该用户详情以保持界面数据一致。两者共同体现后台系统的权限管理和资源维护能力。

7.3 关键技术点

项目中较有代表性的实现包括：基于 localStorage 与路径前缀的路由守卫、Axios 请求/响应拦截器（自动注入 token 请求头、统一解包 { code, msg, data } 响应结构、按状态码分类的 HTTP 错误提示、401 自动跳转登录）、基于请求拦截器的 Mock 数据模拟（mock/index.ts 通过 patternToRegex 将 REST 风格路径如 /post/:id 转换为正则并支持路径参数解析）、ECharts 图表组件集成、fetch + ReadableStream 实现的 SSE/分块流式输出以及基于 Ant Design Vue customRequest 与原生 <input type="file"> 的图片/文件上传处理。这些内容可以作为课程报告中的重点实现说明。

另外，前端对接口参数和返回值进行了统一类型定义（src/api/types.ts 中定义了约 40 个接口），减少了联调过程中的字段歧义；请求层还对业务错误（code !== 1）和 HTTP 错误（4xx/5xx 状态码，按 400/401/403/404/408/500/501/502/503/504 分别给出中文提示）分别处理，提升了异常场景下的用户体验。

7.4 接口与数据流说明

系统的数据流可以概括为“页面事件触发请求 → http.get/post/put/delete 统一封装 → Axios 拦截器注入 token 并解包响应 → 后端返回业务数据 → 页面根据响应刷新视图”。对于普通接口，页面通过 Axios 发送 GET、POST、PUT、DELETE 请求；对于 AI 问答场景，页面绕开 Axios，改用原生 fetch 以流式方式接收后端增量文本，再逐步渲染到聊天窗口中。

为了提升开发效率，项目在本地开发阶段可通过环境变量 VITE\_USE\_MOCK 控制是否启用 mock（当前 .env.development 与 .env.production 均将其设为 false，即默认不启用），mock/index.ts 中目前仅对社区首页信息流接口 GET /post/feed 定义了模拟规则，其余接口（包括全部鉴权接口）均直接请求真实后端，这体现了“核心联调接口优先接入真实服务、边缘展示型接口保留 mock 兜底”的渐进式联调策略。

在权限处理上，前端通过请求拦截器自动在请求头的 token 字段注入身份信息，通过路由守卫控制访问边界，当用户未登录或 token 失效（响应 code 为 401 或 HTTP 状态码为 401）时，会动态 import 路由实例并自动清除 token、跳转到与当前路径匹配的登录页（管理端跳 /admin/login，用户端跳 /login）。这种机制保证了用户端与管理端的访问安全性。

7.5 项目特点与实现亮点

●	双端统一：用户端与管理端共用同一套项目基础能力（同一个 Axios 实例、同一套路由守卫逻辑、同一套颜色变量体系），但在界面风格（Vant 与 Ant Design Vue）和交互方式上做了明显区分，形成了清晰的角色边界。

●	结构化业务：猫咪档案、健康事件、打卡记录和社区帖子都以 api/types.ts 中定义的结构化数据为核心，便于后续统计和扩展。

●	交互完整：从登录注册、列表浏览、详情查看到发布、评论、打卡和后台维护，覆盖了课程设计常见的完整业务链路，共 33 个功能点均可独立验证。

●	技术组合合理：移动端 UI（Vant）、后台 UI（Ant Design Vue）、图表展示（ECharts）和 AI 问答渲染（Marked + 原生 fetch 流式处理）分别选择了适配性更强的技术方案，提高了开发效率。

●	健壮性设计：从 SSE/分块流协议兼容判断，到知识库多种响应形状的兼容解析，再到点赞收藏的乐观更新配合去重保护，代码中多处体现了面向“后端接口尚不稳定”场景的防御性编程思路。

●	开发阶段友好：通过 mock 数据、统一请求封装和类型定义，前端在后端接口未完全稳定时也能持续推进页面开发。

7.6 开发难点与解决方案

在实现过程中，较有代表性的难点主要有以下几类：

难点	具体表现	解决方式（对应代码）

多页面路由管理	用户端和管理端页面数量较多，且需要区分登录态	按场景拆分路由并在 router.beforeEach 中统一使用白名单 + 路径前缀判断

评论树展示与删除	评论支持多层嵌套回复，删除需要联动统计计数	CommentItem 递归组件渲染；countWithChildren / removeCommentById 递归统计与移除

AI 流式输出协议不确定	回复内容需要逐步刷新，且响应可能是 SSE 也可能是普通分块流	先判断 Content-Type，再分别走 SSE 事件解析或原始 chunk 直传两套逻辑

点赞/收藏重复请求	用户可能连续快速点击导致竞态	使用 Set<number> 记录 interacting 状态，请求中禁止重复触发

状态同步类错误干扰体验	重复点赞/收藏会返回业务错误码	响应拦截器识别特定错误文案并静默处理，不弹出 Toast

文件上传	打卡、猫咪照片、知识库都涉及上传，且校验规则不同	分别封装 uploadCheckinPhoto / uploadPhotos / uploadDatasetFile，统一使用 FormData + multipart/form-data

权限控制	不同角色（普通用户/管理员/超管）访问范围不同	使用 token 与登录状态拦截非法访问，401 时自动清理登录态并跳转

统计图表适配	仪表盘需要动态响应布局变化	结合 ECharts 与 window resize 事件调用 chart.resize() 自适应

接口响应结构不稳定	知识库分页结构、数据集列表结构等尚未完全固化	对多种可能的响应形状（array/{rows}/{list}/{data}/{records}）做兼容解析

8.1 页面设计说明

在页面设计上，用户端页面采用卡片、弹层和底部导航组合，尽量减少页面跳转；管理端页面则以表格、筛选栏和弹窗/抽屉表单为主，保证数据处理效率。两类页面虽然风格不同，但都遵循统一的交互原则：操作明确（如危险操作二次确认）、反馈及时（Toast/Message 提示）、层次清晰（弹层/抽屉分层展示详情）。

9  系统测试

系统测试应覆盖功能正确性、异常处理和界面兼容性三个层面。功能测试重点验证登录注册、分页查询、筛选搜索、帖子发布、评论回复、打卡提交以及后台增删改查是否符合预期；异常测试则重点检查空数据、重复操作、超时和未登录访问等场景，其中部分异常处理逻辑（如状态同步类错误静默处理、401 自动跳转）已在 utils/request.ts 中实现，测试时应重点验证这些分支是否按预期触发。测试用例编号与第 5.6 节的功能点编号保持一致，便于交叉核对。

异常场景	处理方式（对应代码）	预期结果

未登录访问受保护页面	router.beforeEach 中检测 token 不存在	拦截并跳转到对应登录页

token 失效（401）	响应拦截器识别 code===401 或 HTTP 401，触发 redirectToLogin	清除登录状态并跳转登录页

表单输入为空	页面内 showToast/message.warning 前置校验	阻止提交并显示校验提示

重复点赞/收藏	响应拦截器匹配“已收藏/未收藏/已点赞/未点赞”关键词	静默处理，不重复弹出错误提示

网络请求失败/超时	Axios 错误拦截器按错误类型区分提示	显示“网络连接异常”或“请求超时”等提示

AI 响应非 SSE 格式	sendMessage 判断 Content-Type 后走分块流分支	仍能正确逐段展示回复内容

普通用户越权访问 /admin 路径	路由守卫未对角色类型做细分校验（仅校验 token 是否存在）	建议后续在守卫中补充 userType 校验

在系统设计阶段，本报告采用了规范的软件工程方法：绘制了系统架构图与请求处理流程图，梳理了 4 张类图展现模块间的调用与复用关系，为全部 33 个功能点撰写了包含前置条件、后置条件、基本流程与备选/异常流程的详细用例规约，并为其中 17 个核心场景绘制了时序图，力求以“图文对照代码”的方式完整呈现系统的设计脉络。

从课程设计报告的角度看，本项目的价值不仅在于功能较全面，更在于它覆盖了从需求分析、系统设计到实现和测试的完整开发过程，并且报告中的每一项功能说明、每一张时序图和类图都能在仓库代码中找到对应实现，能够较好地体现 Web 前端应用开发的整体方法与工程素养。

12.1 不足与展望

●	后端接口仍可进一步标准化，尤其是数据结构和错误码规范（如知识库分页结构目前依赖前端做多形状兼容解析）。

●	路由守卫目前仅校验是否登录，未对角色类型（普通用户/管理员）做细粒度区分，存在被绕过访问管理端路由的风险，属于优先级较高的改进项。

●	AI 问答部分还可以增加更强的上下文记忆和多轮对话能力，目前会话历史仅通过 getSessionDetail 一次性拉取展示。

●	管理端的统计内容可以继续扩展，例如增加更多维度的图表和数据筛选（当前仪表盘仅覆盖用户增长与品种分布两类图表）。

●	如果后续接入真实数据库，还可以补充更完整的数据备份、审计和权限控制机制，并将本报告第6章的逻辑表结构落地为真实的物理数据库。

总体来看，本项目已经具备了一个课程设计作品应有的完整度，后续优化主要集中在工程质量、权限细粒度控制和数据真实性三个方向。

13  运行环境与启动说明

13.1 运行环境

本项目以 Web 前端课程设计为主，当前报告对应的前端部分运行环境建议如下（结合 package.json 与 pnpm-lock.yaml 的实际依赖版本要求）：

●	操作系统：Windows 10 / Windows 11（macOS / Linux 同样适用）

●	开发工具：Visual Studio Code（建议安装 Vue.volar 扩展，见 .vscode/extensions.json 推荐配置）

●	前端运行时：Node.js（需满足 Vite 8 与相关依赖的版本要求）

●	包管理工具：pnpm（仓库提供 pnpm-lock.yaml），也可使用 npm

●	浏览器：Chrome、Edge 等现代浏览器

13.2 前端启动方式

根据 frontend/package.json 中定义的脚本，前端可通过以下命令启动和构建：

cd frontend

pnpm install

pnpm dev

如果需要打包发布，可执行：

pnpm build

开发服务器默认运行在 5173 端口（vite.config.ts 中配置了 host: '0.0.0.0'），并通过代理将 /api 前缀的请求转发到后端服务地址，同时自动去除 /api 前缀；如需切换后端地址，可修改 vite.config.ts 中 server.proxy\['/api'].target 配置项，或通过 .env.development / .env.production 中的 VITE\_API\_BASE\_URL 调整接口基础路径。

13.3 使用说明

启动后，用户可先访问登录（/login）或注册（/register）页面完成身份认证，再进入用户端主页面（/archive 等）进行猫咪档案浏览、社区互动、AI 问答和猫咪打卡等操作。管理员则可通过后台登录页（/admin/login）进入管理端，完成仪表盘查看、猫咪档案维护、健康事件管理、知识库维护和用户管理等任务。若需要在无后端环境下先行体验社区首页信息流效果，可将 .env.development 中的 VITE\_USE\_MOCK 设置为 true 以启用 mock/index.ts 中定义的模拟数据。
